# Alembic Migrations

**When developing locally, always prompt the user before creating an Alembic migration.**

Alembic should be treated as a data-preservation tool, not a default local-development workflow. Prefer using it when updating a live system that already contains real data which must survive schema changes.

## Rules

### 1. Local development: ask first

If the task is local development, first deployment, greenfield schema work, or a disposable environment:

- **Do not create an Alembic migration without explicit user approval.**
- Ask whether the user wants a migration at all before running `alembic revision`, `alembic upgrade`, or wiring migration scripts into the workflow.
- Do not assume that "proper database changes" means Alembic is wanted.

### 2. Default away from Alembic on first deployments

For first deployments and other cases where there is no real data to preserve:

- Prefer simpler setup paths such as rebuilding the schema, reinitializing the database, or updating bootstrap SQL/models directly.
- Do not introduce Alembic just to satisfy framework convention.
- If Alembic is being considered anyway, stop and explain that it is usually unnecessary for an initial deployment with disposable data.

### 3. Use Alembic when data preservation matters

Alembic is appropriate when updating an existing live system where schema changes must preserve real data.

Typical signals:

- Production or shared environments already contain user data
- A schema change must be rolled out without dropping tables
- Data transformations or backfills are required during upgrade

In those cases:

- Create explicit, reviewable migrations
- Inspect autogenerated output before accepting it
- Prefer reversible or carefully documented upgrade paths

## Prompting Guidance

When Alembic comes up in local development, ask a direct question such as:

> "This looks like local or first-deployment work. Do you want an Alembic migration here, or would you rather keep this as a schema/bootstrap change for now?"

## Do Not

- Do not generate local migrations without asking
- Do not present Alembic as mandatory for first deployment
- Do not hide migration creation inside setup scripts or app startup for local environments
